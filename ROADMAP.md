# AirControl Roadmap

Этот roadmap нужен, чтобы проект развивался как ассистивный инструмент, а не
как разовая демонстрация. Приоритеты ниже идут от пользовательской пользы:
сначала надёжность и доступность, потом новые способы управления.

## Принципы

- Пользователь не обязан пользоваться консолью.
- Первый запуск должен быть безопасным: без случайных кликов и клавиш.
- Любая ошибка должна давать понятный следующий шаг или ZIP-отчёт.
- Linux/Wayland проблемы нужно диагностировать явно, а не скрывать.
- Новые жесты не должны ухудшать управление для людей с ограниченной моторикой.
- Сборки должны проверяться на чистом checkout, а не только на машине разработчика.

- [x] Ассистивный профиль с dwell-click.
- [x] Dwell-click profiles: быстрый, обычный, устойчивый.
- [x] Пауза после dwell-click в профилях, чтобы снизить повторные клики при треморе.
- [x] Режим `один жест`: наведение и dwell-click без щипков/сложных жестов.

## 1. Надёжность Управления

Пользовательская проблема: рука видна, но курсор/клики не работают или работают
нестабильно.

- Input wizard в GUI: отдельный экран `Камера -> Рука -> Ввод -> Готово`.
- Безопасный тест движения курсора на 1 px из стартового окна и preflight перед control.
- Отдельные статусы для камеры, FPS, распознавания руки и input backend.
- Автоматическое предложение Xorg/ydotool для Linux, если обнаружен Wayland.
- Настройки чувствительности и dwell-click крупными кнопками без клавиатуры.
- Сохранение нескольких пользовательских профилей.

## 2. Доступность Для Маломобильных Пользователей

Пользовательская проблема: щипок, быстрые жесты или точное движение рукой могут
быть недоступны.

- Dwell-click profiles: медленный, обычный, быстрый.
- Большая экранная панель действий: клик, двойной клик, ПКМ, скролл, пауза.
- Режим `минимальное движение`: меньшая активная зона и более сильное сглаживание.
- Расширить режим `один жест`: отдельные крупные действия и настройки под тремор.
- Ручная настройка паузы после клика для отдельных пользовательских профилей.
- Настраиваемый размер зоны удержания для пользователей с тремором.

## 3. Новые Способы Ввода

Новые возможности должны быть опциональными и отключаемыми.

- Head tracking как fallback, если кисть трудно удерживать в кадре.
- Голосовые команды в офлайн-режиме через Vosk.
- Экранная клавиатура с dwell selection.
- Радиальное меню команд по удержанию открытой ладони.
- Макросы пользователя: открыть приложение, нажать сочетание клавиш, вставить текст.
- Переключение между несколькими мониторами.

## 4. Персонализация Жестов

Пользовательская проблема: стандартные жесты подходят не всем.

- GUI-мастер записи жеста без терминала.
- Персональный ML-профиль на 5-10 примерах жеста.
- Импорт/экспорт профиля пользователя.
- Оценка уверенности жеста в интерфейсе.
- Конфликт-детектор: предупреждать, если два жеста слишком похожи.

## 5. Установщики И Обновления

Пользовательская проблема: человек не должен знать, что такое Python.

- Windows installer с ярлыком и uninstall.
- macOS `.app` в zip, позже подпись и notarization.
- Linux `.deb` и AppImage.
- Проверка обновлений без автоматической установки.
- Страница Releases с понятными файлами для каждой ОС.
- Краткая инструкция первого запуска внутри каждого архива.

## 6. Качество И Метрики

Без метрик невозможно понять, стало лучше или хуже.

- Метрика FPS и slow detection в support ZIP.
- Минимальный performance benchmark на синтетических кадрах.
- Тесты распознавания жестов на стабильность порогов.
- Regression tests для hotkeys в русской и английской раскладке.
- Release checklist перед тегом.
- Таблица известных проблем по ОС.

## 7. Будущие Интеграции

Эти направления не нужны для первого релиза, но могут сделать проект долгим.

- Плагинная система действий.
- Интеграция с системными accessibility API.
- Нативные helper-модули только там, где они дают реальную пользу: диагностика,
  accessibility bridge, tray/auto-update или системная установка.
- Пресеты для браузера, текстового редактора, презентаций и видеозвонков.
- Мини-сервис для автозапуска AirControl при входе в систему.
- Лёгкая оболочка на Tauri/Rust только если понадобится tray/auto-update.

## Что Не Делаем Сейчас

- Не переписываем ядро на другой язык ради самого переписывания.
- Не добавляем жесты, которые повышают риск случайного клика.
- Не делаем облачную телеметрию по умолчанию.
- Не требуем от пользователя терминал для базовой диагностики.
- Не обещаем полную работу Wayland без настройки input backend.

## Release Gate

Релиз можно считать готовым, когда выполнены все пункты:

- CI зелёный.
- Build AirControl зелёный на Windows, macOS и Linux.
- Артефакты скачиваются и проходят `tools/verify_release_artifacts.py`.
- Первый запуск открывает стартовое окно без консоли.
- Safe/View режим не отправляет input events.
- Assistive режим работает на машине с камерой.
- Linux tester report содержит понятный вывод по Xorg/Wayland/input backend.
- README и BUILD соответствуют фактическим артефактам.
